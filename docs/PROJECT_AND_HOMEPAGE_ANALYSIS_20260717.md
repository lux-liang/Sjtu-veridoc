# Sjtu-veridoc 项目剖析与首页改版说明

更新时间：2026-07-17

## 一、项目现状结论

Sjtu-veridoc 已经形成“PDF 结构、图像取证、印章、OCR/文本、业务逻辑、Qwen-VL 复核”的多证据体系。当前最成熟、最有真实迁移价值的能力是业务字段勾稽；传统像素取证在同源扫描文档基准上接近随机，不能作为主要卖点。

线上页面由一个 Python `BaseHTTP` 自定义服务提供，当前源码只存在于部署机 `/root/Liangjialiang2`，没有纳入 GitHub 仓库。仓库中的 README 也明确指出 `app.py` 缺失。这是当前最大的工程风险：页面无法代码审查、无法复现、无法通过 Git 正常发布和回滚。

### 已有优势

1. `v3_sign_corrected` 修复了旧评分方向倒挂，理由代码可解释、可审计。
2. 已建立 provenance-matched 同源基准，能够识别“高 AUC 是来源泄漏”这一常见陷阱。
3. 发票金额勾稽和银行余额连续性在受控同源基准上达到 AUC 1.0。
4. Qwen-VL 是独立证据通道，原因带 `qwen:` 前缀，没有强行覆盖规则证据。
5. 扫描件上下文已有降权逻辑，方向正确。

### 当前关键问题

1. **首页看不到检测指标。** 当前只展示样本数、风险分布和风险原因，用户无法知道标签集上的 Precision、Recall、F1。
2. **指标容易被误读。** 线上 537 份带标签样本以风险分 `≥25` 判疑似虚假时，Accuracy、Precision、Recall、F1 均为 100%，但 279 份假样本中有 276 份带显式 `synthetic/edited/training` 标记；该数字不能代表真实新材料的泛化性能。
3. **“检测原理”只有技术视角。** PDF、PS、印章、OCR、业务逻辑是证据维度，不是客户购买和验收的业务产品线。
4. **红章路线不能覆盖黑白复印件。** 当前红色阈值和红色连通域是颜色依赖方法，灰度扫描后信号直接消失。
5. **默认文档接口只返回 500 行。** `/api/documents` 的 `count` 为 542，但默认 `rows` 只有 500；聚合评估必须按标签分别取数或由后端直接返回指标。
6. **上传检测与平台完整能力不一致。** 页面明确写的是“纯后端对象结构扫描”，上传件尚未走 OCR、视觉、业务规则和印章全链路。
7. **页面原理权重存在陈旧风险。** 页面文案中的部分权重与 `src/build_combined_risk.py` 的 v3 实际权重不完全一致，后续应由代码配置生成说明。

## 二、首页“带标签检测表现”口径

### 当前线上聚合结果

判定规则：`combined_risk_score >= 25` 为疑似虚假。

| 指标 | 数值 |
|---|---:|
| 带标签样本 | 537 |
| TP | 279 |
| FP | 0 |
| TN | 258 |
| FN | 0 |
| Accuracy | 100.0% |
| Precision | 100.0% |
| Recall | 100.0% |
| F1 | 100.0% |

必须与指标同时展示的限制：

- 276/279 个假样本含显式伪造/训练标记。
- 无显式标记的假样本只有 3 份，规模不足以估计真实泛化性能。
- 因此标题应写“带标签样本检测表现”或“当前标签集内表现”，不能写“系统真实准确率 100%”。

首页补丁采用页面实时聚合：分别请求 `label=fake` 和 `label=normal`，避免默认 500 行上限造成漏算，并显示混淆矩阵和样本覆盖情况。

## 三、五大业务产品线

### 1. PDF 篡改检测算法开发

虚假类型：

- 局部贴图、遮盖、改金额或改姓名。
- Word 转 PDF、图片转 PDF、打印后扫描重制。
- 增量更新、二次保存、注释或嵌入对象。
- 页面删除、插入、替换、跨来源拼接。

技术路线：

`PDF 对象/字节结构 → 多页渲染 → 文本层与 OCR 对齐 → 跨页元数据/版式一致性 → 证据融合`

当前基础较好，但需补充多页抽样、页面来源一致性和 Word/图片重导出来源分类。注意“Word 转 PDF”本身不等于造假，只能作为来源或复核证据，必须与内容异常交叉印证。

### 2. 虚假印章算法开发

虚假类型：

- PNG/JPG 透明贴章。
- 从其他材料复制同一印章。
- 矢量重绘、字体重排或模板生成印章。
- 黑白扫描章、复印章、低对比度章。
- 印章主体名称或盖章位置与业务材料不匹配。

推荐路线：

`印章候选定位 → 抠图/分割 → 几何与纹理 → 环形文字展平/OCR → 相似印章检索 → 文档主体与位置勾稽`

#### 黑白扫描件专项

不建议把多模态大模型作为唯一定位器。它适合快速验证、生成弱标注和处理困难样本，但坐标稳定性、费用、时延和金融隐私都不适合主生产链。建议：

1. 用本地目标检测器或版面模型定位印章候选；多模态大模型作为兜底和弱标注工具。
2. 裁剪时保留 15%–25% 周边上下文，避免只看章而丢失纸张融合和盖章位置信息。
3. 用分割模型或灰度/边缘方法扣出印章，不能再依赖红色阈值。
4. 用圆/椭圆拟合、径向投影、极坐标展开识别环形文字和中心图案。
5. 分析复印半色调网点、边缘锐化、灰度层次、局部噪声、纸张纹理连续性。
6. OCR 提取机构名，与合同甲乙方、发票销售方、报告出具机构等实体做语义勾稽。
7. 建立印章向量库，发现同一印章图片在不同材料、不同机构之间复用。

评测必须分层报告：红色原章、红色扫描章、黑白扫描章、复印章、低分辨率章；同时报告候选定位 mAP、印章 OCR 字符准确率和文档级 Precision/Recall/F1。

### 3. PS 发票检测算法开发

虚假类型：金额/税额/合计修改，购销方或税号替换，发票号码/日期修改，二维码替换，印章/签名贴图，截图重存。

技术路线：

`发票版式识别 → 字段 OCR 与坐标 → 金额勾稽/二维码交叉验证 → 字体背景行距 → PDF 图层与局部取证 → 多证据融合`

其中金额勾稽已经具备高价值；下一步应优先做二维码内容交叉校验和字段区域模板，而不是继续堆全图 ELA 均值。

### 4. 征信报告篡改检测算法开发

虚假类型：身份信息修改，账户与逾期记录修改，遮盖/删除不良记录，余额或授信额度修改，页面删插和拼接。

技术路线：

`章节/版式解析 → OCR 坐标 → 字体/背景/行距与表格线 → 页码/报告编号/页眉页脚 → 跨页业务语义 → 人工复核`

征信报告多为扫描件，不能把“无文本层、OCR 低置信、字段缺失”直接当成高风险。应先做扫描质量分层和模板区域对齐，再判局部异常。

### 5. 相似图片检测算法开发

虚假类型：完全重复、压缩缩放后的近重复、裁剪旋转加水印、局部复制覆盖、印章/签名跨材料复用。

技术路线：

`pHash 粗召回 → 对比学习全局向量 → ANN 向量库 → 局部特征/密集向量 → 几何验证 → 分场景阈值校准`

全局相似度只能回答“整体像不像”；局部贴图必须增加区域级匹配和几何验证，否则会被文档模板、表格线和重复文字造成大量误报。

## 四、建议优先级

### 2026-07-17 已落地进展

- 综合风险汇总已原生输出带标签混淆矩阵、Accuracy、Precision、Recall、F1 与 marker 依赖审计，首页补丁会优先读取该产物。
- 已实现不依赖红色的印章候选定位，可输出黑白/复印章候选坐标，并保存上下文裁剪与 OCR 增强图。
- 已实现本地印章 OCR 后处理：外环极坐标展开、多视图 Tesseract、印章文字与甲乙方/购销方/机构主体相似匹配；未安装 OCR 时不会阻断主流程。
- Qwen-VL 可选通道可返回结构化印章坐标、颜色、文字与贴图怀疑分，作为本地定位器的困难样本兜底和训练弱标注来源。
- 视觉/印章检测已从固定第一页扩展为前 N 页模式，远程迭代默认扫描 3 页并记录最佳候选页码。
- 已建立 120 份零 PII 合成定位基准：Recall 98.75%、Precision 79.80%、F1 88.27%、检测成功样本平均 IoU 0.596。
- 圆形 Logo 困难负样本全部被召回，证明当前阶段解决的是“章形区域召回”，不是最终“印章语义识别”。下一轮应以印章 OCR、机构实体匹配和盖章位置语义降低这类误报。
- 改版已部署至线上 3002 端口。实际部署目录为 `/home/lighthouse/sjtu_material_visual`；systemd 工作目录和启动路径已从失效的 `/opt/sjtu_material_visual` 修正，服务由遗留手工进程切换为稳定受管运行。
- 生产视觉特征已重算：234 份材料、296 页、0 错误、203 个候选；OCR 成功 202 个。经置信度约束后主体不匹配复核只剩 2 条，正常样本为 0。
