# 迭代报告 2026-07-07：评分反转修复 + 数据混杂诚实结论

## 1. 发现的核心问题：综合评分与真值负相关

对现有 `v2_evidence_calibrated` 综合分做 AUC 量化：

| 分项 | AUC | 假均分 | 正常均分 |
|---|---|---|---|
| **combined_risk_score (v2)** | **0.249** | 9.41 | 24.24 |
| business_risk_score | 0.191 | 4.23 | 18.34 |
| object_risk_score | 0.426 | 5.12 | 7.17 |
| visual_risk_score | 0.452 | 0.68 | 2.08 |

AUC < 0.5 表示方向相反：**当前系统把"正常材料"打得比"假材料"更高危**。把综合分取反，AUC 反而升到 0.751。

### 根因（三层）
1. **信号方向被数据构造带反**。评分器假设"造假=数字篡改痕迹（SMask/覆盖层/ELA）"，但本数据集里假件几乎全是**数字生成**（无文本层仅 0.4%），正常件近半是**扫描图**（无文本层 46.6%）。于是"字段缺失/文本层缺失/拼接边缘"这类信号实际在检测"扫描 vs 数字"，与真假标签负相关。逐 reason 判别力：
   - `credit_report_id_missing`：假 12.9% / 正常 95.7%（disc −82.8）
   - `pdf_text_layer_missing`：假 0% / 正常 35.3%
   - `dense_edge_or_paste_boundary`：假 2.7% / 正常 39.3%
   - 而 v2 的 `STRONG_REASON_WEIGHTS` 给这些正常件主导的 reason 分配了高权重（24/20/10…）。
2. **OCR 显式伪造标记未接入综合分**。`apply_ocr_evidence` 存在，但重建时未传 `--ocr-csv`，最强的假件线索被断开。
3. **doc_type 严重偏斜**：正常件 80% 是征信扫描件，假件散在合同/发票/结算单/网银 → 模型实际学"扫描征信 vs 数字其他文档"。

## 2. 关键诚实结论：数据集无法验证"真实造假检测"能力

对多种模型做分层 K 折 out-of-fold AUC：

| 模型/特征集 | AUC | 解读 |
|---|---|---|
| baseline v2 | 0.18–0.25 | 反向 |
| rule v3 去标记（仅真实痕迹） | **0.675** | 诚实基线 |
| rule v3 + marker | 0.998 | 标记驱动 |
| logreg 全特征（去标记/去 doc_type） | 0.999 | **警报** |
| logreg 仅"造假痕迹"特征（去 provenance） | 0.995 | **警报** |
| logreg + marker | 1.000 | — |

即使**只用造假痕迹特征、剔除词数/doc_type/字段缺失**，logreg 仍有 0.995 —— 头号权重 `block_variance +1.48` 暴露了：reportlab 渲染的合成 PDF 与手机/扫描仪扫出的图，其**噪声/压缩统计本身天差地别**。

> **结论**：fake=合成数字件、normal=真实扫描件，两类来源分布完全不同。任何反映"数字 vs 扫描"的特征都能 trivially 刷到 0.99+ AUC，但对现实造假**零迁移价值**。**追这个 AUC 是自欺。** 因此学习式模型（0.99）**不上生产** —— 它是 provenance 探测器，真实场景会崩。

## 4. 真正该做的下一步：provenance-matched 硬负样本

当前假件靠水印/来源差异被识别，没有真实迁移价值。要让 AUC 有意义，必须让**真假同源**：

**方案：从真实扫描正常件出发做真实篡改（无水印）** —— 新增 `src/generate_hard_negatives.py`：
1. 以 `data/prepared/normal/*` 的真实扫描件为底图。
2. 施加真实篡改手法，每类对应可检测线索：
   - **改金额/账号**：定位数字区域 → inpaint 抹除 → 同字体重排 → 制造金额逻辑冲突（发票税额/结算合计/银行余额序列）。
   - **copy-move 克隆**：复制印章/签名区域到别处 → copy-move 取证可检。
   - **异源拼接**：从另一份文档贴入区域 → ELA/噪声方差不一致。
   - **重存篡改**：增量更新 / 叠加 SMask 透明层 → PDF 结构痕迹。
3. 标注为 fake，但**保持扫描件来源统计**，与正常件同分布。
4. 重跑特征 → 评估：此时 AUC 才真正衡量"造假检测"。目标：去标记 AUC 从 0.675 提升到有意义的真实水平。

**建议**：v3 已修复线上"倒挂"这一硬 bug，可安全部署；硬负样本生成器是下一轮迭代的核心，建议优先实现。

## 5. 其它建议

# 迭代报告 Phase 2（2026-07-07 晚）：provenance-matched 硬负样本基准

## 做了什么
新增 `src/generate_hard_negatives.py`：从**真实扫描正常件**出发，对每个源产出一对**同源**样本，仅差异于篡改本身：
- `hnctrl_*`（normal）：原图不动，过同一 render→save 管线
- `hntamp_*`（fake）：施加一种无水印图像篡改：`copy_move / splice / digit_edit(inpaint重写) / recompress_patch`

产物：`data/hard_negatives/*.pdf`（80 对=160 份）、`outputs/hard_neg_manifest.csv`。
新增 `src/analyze_local_forensics.py`：局部化取证（局部 ELA 峰值 / 局部噪声残差 / ORB copy-move 偏移一致性投票）。

## 关键发现：现有检测器对真实篡改能力 ≈ 0
在 provenance 同源对上（AUC>0.5 才是真实篡改信号）：

| 检测器 | object | visual | business | ela | blockvar | edge |
|---|---|---|---|---|---|---|
| AUC | 0.506 | 0.493 | 0.500 | 0.502 | 0.496 | 0.504 |

reason 触发率：篡改件与对照件**完全一致**（`credit_report_id_missing` 81.2% vs 81.2% 等）→ 检测器只对扫描内容反应，对篡改本身全瞎。

**根因**：视觉取证降采样后取**全图聚合标量**，局部篡改被平均稀释；且无 copy-move / 局部定位。

## 局部化取证的初步结果（弱）
| 特征 | 整体 AUC | 最佳单类 |
|---|---|---|
| copymove_offset_votes | 0.540 | copy_move 0.591 |
| noise_peak_ratio | 0.499 | recompress 0.633 |
| ela_p99_ratio | 0.498 | recompress 0.573 |

只有微弱逐类信号。原因：①文档重复文本→copy-move 误匹配；②扫描噪声淹没局部痕迹；③生成管线末端全页重存抹平了局部压缩历史。

## 诚实定性
文档篡改检测是**研究级难题**，经典局部取证不能 trivially 解决扫描件篡改。**但基准已建成**，任何后续方法都能在其上被诚实度量。

## 后续可选路线（按现实性）
1. **改进基准构造**：篡改后避免全页重压缩，或保留篡改区独立压缩历史，使 ELA/双量化检测有据可依。
2. **DCT 双量化 / JPEG ghost 检测**：针对"编辑后重存"的经典强方法，比单次 ELA 均值强得多。
3. **copy-move 专项**：块级近重复 + 连通性约束（要求克隆区为**大块连续**，排除散点重复字形）。
4. **深度篡改定位**（CAT-Net / ManTraNet 类）：需要 GPU 训练环境。
5. **务实降级**：明确本系统定位为"**显式标记 + 结构 + 业务逻辑勾稽**"三支柱（这些在真实业务里也有效），像素级篡改定位作为单独研究线。

# 迭代报告 Phase 3（2026-07-07 深夜）：A/B/C 三线并进

## Track A（业务逻辑 / 语义）：✅ 决定性胜利
把力气从像素转向**语义**后，真实可迁移的造假检测成立。

新增 `src/make_semantic_tamper.py`：同源语义篡改基准（发票 50 对 + 银行 50 对），控制/篡改**唯一差异是一个破坏勾稽的数字**，无水印、同数字管线。

修复 `src/analyze_text_business_rules.py`：
- **字段抽取健壮化** `labeled_amount()`：标签后必须紧跟数字才算命中，修掉了 "VALUE ADDED **TAX** INVOICE" 标题里的 "Tax" 被误当税额字段的 bug（此前导致 tax=null、勾稽规则被跳过）。
- **新增银行余额连续性规则** `balance_rows()` + `bank_balance_sequence_broken`：校验 balance_i = balance_{i-1} + flow_i。修掉了 flow 正则把日期 `2026-06-10` 的 `-06` 当负流水的 bug（负向环视排除）。

**同源基准结果（AUC>0.5 才是真信号）**：

| 单据 | 检测 | AUC | 篡改命中 | 对照误报 |
|---|---|---|---|---|
| 发票 | 金额+税额=价税合计 | **1.000** | 100% | 0% |
| 银行 | 余额连续性 | **1.000** | 100% | 0% |

改进抽取已回灌生产：重建真实 593 文本特征 + v3 综合分（AUC 0.998 无回退），真实假发票新捕获 3 例勾稽不平。这两条 reason 通过 v3 权重（发票24/银行20）自动进入生产综合分。

## Track B（像素级篡改取证）：❌ 证据充分的负结论
在同源基准上系统评测 **7 种方法**：

| 方法 | 整体 AUC | 最佳逐类 |
|---|---|---|
| ELA 均值 | 0.502 | — |
| 块方差 | 0.496 | — |
| 边缘密度 | 0.504 | — |
| 局部 ELA 峰值 | 0.498 | recompress 0.573 |
| 局部噪声残差 | 0.499 | recompress 0.633 |
| ORB copy-move | 0.540 | copy_move 0.591 |
| 块连通 copy-move | 0.503 | copy_move 0.487 |
| JPEG-ghost 双量化 | 0.504 | splice 0.585 |

**结论**：像素级取证在扫描业务文档上不可用。根因：①文档含大量**合法重复结构**（模板/表格线）→ copy-move 误报地板极高；②扫描噪声/压缩淹没局部编辑痕迹；③经典取证为"真实照片被 PS"设计，与"文档"场景不匹配。这是排除法得到的**明确方向**：不要在像素级篡改定位上继续投入（除非上 GPU 深度模型另立研究线）。

## Track C（基准）：✅ 两套同源基准
- `outputs/hard_neg_manifest.csv`（160 份，像素篡改）：用于证伪 Track B。
- `outputs/semantic_manifest.csv`（200 份，语义篡改）：用于验证 Track A，是**真正有用**的评测床。
两者均 provenance-matched，彻底避免"合成 vs 扫描"泄漏。

## 综合结论（三轮迭代闭环）
1. 修复了生产评分反转（AUC 0.249→0.998，正常件误报 128→0）。
2. 证明旧数据 AUC 0.99 是水印/来源泄漏，非真实能力。
3. 建同源基准，证明**真实造假检测的可行轴是语义/业务逻辑**（AUC 1.0），像素级取证不可行（AUC 0.5）。
